## Overlay Contrast Matrix

This phase introduces a second full-screen overlay. Both overlays must be visually distinct.

| Overlay | Trigger | Background | Text | z-index |
|---------|---------|-----------|------|---------|
| Data load (`LoadingSpinner`) | `loading === true` (network fetch) | `bg-white/90 backdrop-blur-sm` | "Chargement des prix..." (`text-gray-600 text-sm font-medium`) | `z-[2000]` |
| Filter overlay (new) | `filterPending === true` (filter change) | `bg-black/40` | none | `z-[2000]` |

Both overlays are `absolute inset-0`. They are mutually exclusive in normal flow (data loading completes before the user can change filters), but the z-index is identical so stacking order is irrelevant.

Source for data-load overlay: `src/components/LoadingSpinner.tsx` (pre-existing, do not modify).
Source for decisions: `03-CONTEXT.md` D-01, D-02.

## Component Specification

### FilterOverlay (new component)

Implement as a standalone component — do NOT add a `variant` prop to `LoadingSpinner`.
Rationale: keeps both overlays independently maintainable and avoids coupling network-load semantics with filter-change semantics.

**File:** `src/components/FilterOverlay.tsx`

**Anatomy:**

```
<div> ← overlay root: absolute inset-0 z-[2000], bg-black/40
  <div> ← spinner: w-10 h-10, border-4, rounded-full, animate-spin
         border-white/30 border-t-white
```

**Classes (exact):**

| Element | Tailwind classes |
|---------|-----------------|
| Overlay root | `absolute inset-0 z-[2000] flex items-center justify-center bg-black/40` |
| Spinner | `w-10 h-10 border-4 border-white/30 border-t-white rounded-full animate-spin` |

**Dimensions:**
- Spinner: 40px × 40px (`w-10 h-10`) — slightly smaller than the data-load spinner (48px `w-12 h-12`) to reinforce visual distinction
- Touch target: not applicable — overlay is not interactive

**No text label.** Source: D-02.

**No backdrop-blur.** The dark semi-opaque background provides sufficient separation without blur; blur on a dark overlay adds visual noise and is inconsistent with the existing filter panel aesthetic.

### Transition (optional, Claude's discretion)

Apply a fade-in/fade-out on mount/unmount using Tailwind's transition utilities, consistent with the established `transition-opacity duration-300 ease-in-out` pattern from `HamburgerMenu.tsx`.

Recommended approach: CSS opacity transition controlled by a mounted boolean, keeping the overlay always-rendered-but-invisible pattern (same as `HamburgerMenu`) to enable the exit animation.

| State | Classes |
|-------|---------|
| Visible | `opacity-100 pointer-events-auto` |
| Hidden | `opacity-0 pointer-events-none` |
| Transition | `transition-opacity duration-150 ease-in-out` |

Duration 150ms matches the minimum display time guarantee — the fade-out begins exactly when the guard expires, so the visual transition coincides with the minimum display floor.

## Interaction Contract

### Trigger

The filter overlay triggers on every change to `filterState` in `App.tsx` — both:
- Fuel type selection changes (`selectedFuelType`)
- Brand selection changes (`companies`)

Source: D-04.

### Lifecycle

```
filterState changes
  → set filterPending = true   (overlay becomes visible)
  → record overlayStartTime = Date.now()
  → requestAnimationFrame / setTimeout(0) → run filterStations()
  → setFilteredStations(result)
  → compute elapsed = Date.now() - overlayStartTime
  → remaining = Math.max(0, 150 - elapsed)
  → setTimeout(() => set filterPending = false, remaining)
```

Minimum display time: **150ms**. Source: D-03.

### Rapid change handling

If `filterState` changes again while the overlay is already visible:
- Reset `overlayStartTime` to `Date.now()`
- Re-run filter with new state
- The 150ms guard restarts from the most recent change

Source: `03-CONTEXT.md` Specifics — "if the user rapidly changes filters while an overlay is active, the timeout should reset."

Implementation note: use a `useRef` to track the timeout ID and clear it before setting a new one. This avoids stacking multiple `setTimeout` callbacks.

### Integration point

`App.tsx` `useEffect([stations, filterState])` — the overlay show/hide lifecycle wraps this existing effect. No changes to `FilterPanel.tsx` or `filterUtils.ts` are needed.

Source: `03-CONTEXT.md` Integration Points.

## Accessibility Notes

- The filter overlay must carry `aria-hidden="true"` — it is a decorative loading indicator, not an interactive element.
- Consider adding `role="status"` with `aria-label="Filtrage en cours"` on the overlay root if screen reader announcement of filter activity is desired. This is left to implementer discretion — it is not a blocking requirement for this phase.
- The spinner itself (`animate-spin`) has no accessible name — acceptable since the overlay is non-interactive and the user's action (changing a filter) already provides context.
